## 2026-04-02 串口参数下发：频率可生效、路由偶发失败

### 1) 现象

- 工控屏修改点频（示例 1580MHz）时，主控可正常接收并生效。
- 工控屏修改路由（0x12310000）时，主控日志偶发报错：

```text
[UART DEBUG] Read 14 bytes from UART
[UART ERROR] CRC Failed or Bad Tail. Dropping fake header.
```

- 用户抓到的典型帧如下：

```text
D5 5D 01 FF 09 12 35 10 00 00 18 1B E0 31 82 5D D5
D5 5D 01 FF 06 12 31 00 00 01 F9 91 5D D5
```

- 结果表现为“串口看起来收到了数据，但配置没有下发到主控业务处理”。

### 2) 原因

- 根因不在 HMI 封包格式，也不在路由地址定义。
- 根因在主控串口接收拼包逻辑。修复前关键代码如下：

```c
char rx_buf[1024];
char temp_buf[1024];
char rx_len = 0;

len = read(ui_Fd, rx_buf, sizeof(rx_buf));
...
memcpy(rx_buf + rx_len, temp_buf, len);
rx_len += len;
```

- 脏数据覆盖是这样发生的：

1. `read()` 直接把新串口数据写进 `rx_buf[0..len-1]`，而不是写进临时缓冲区。
2. 如果上一轮已有半包（`rx_len > 0`），这次 `read()` 会从 `rx_buf` 起始位置覆盖旧半包前部，破坏原始帧连续性。
3. 紧接着 `memcpy(rx_buf + rx_len, temp_buf, len)` 又把 `temp_buf` 拷贝到尾部，但 `temp_buf` 并没有被 `read()` 填充，内容是未初始化栈数据（脏数据）。
4. 结果是 `rx_buf` 内部变成“部分新数据 + 被覆盖旧数据 + 未初始化垃圾数据”的混合体。
5. 后续按 `cmd_len` 组帧并做 CRC 校验时，参与计算的字节序列已被污染，因此会出现随机 `CRC Failed or Bad Tail`。

- 另外，原实现里接收缓存使用 `char`，在 CRC 字节拼接场景下可能触发符号扩展，进一步放大偶发误判概率。

### 3) 怎么排查

- 第一步：核对两端协议口径是否一致。
- HMI 发送格式：`cmd=0x01`、`msgType=0xFF`、`payload=[len(1)+addr(4)+value(N)]`，CRC 为 Modbus/A001 并高低字节交换。
- 主控解析格式：按 `cmd_len = rx_buf[4]`，整帧长度 `expected_total_len = 8 + cmd_len`，CRC 区间从 `cmd` 到 `payload` 末尾。
- 第二步：对用户抓包做离线 CRC 复算。
- 两帧 CRC 均正确（频率帧 `0x3182`，路由帧 `0xF991`），说明发送端报文本身没问题。
- 第三步：检查主控收包缓存流程。
- 发现读取后拼接缓存代码异常：读取目标与拼接源缓冲区不一致，导致帧内容被污染。

### 4) 最后修改

- 修改文件：`projects/ui_practice_test/mgmt-ctrl/ui_get.c`
- 修改点如下：

1. `read()` 改为读入 `temp_buf`，再追加到 `rx_buf`。
2. `rx_buf` 类型改为 `uint8_t`，`rx_len` 改为 `int`。
3. CRC 接收值拼接改为显式 `uint16_t` 强制转换，避免符号扩展。

- 关键修复后逻辑：

```c
len = read(ui_Fd, temp_buf, sizeof(temp_buf));
...
memcpy(rx_buf + rx_len, temp_buf, (size_t)len);
...
uint16_t recv_crc = ((uint16_t)rx_buf[expected_total_len - 4] << 8) |
					(uint16_t)rx_buf[expected_total_len - 3];
```

- 修复后已重新编译 `ui_practice` 并部署到 NFS 目录用于联调复测。

## 2026-04-02 路由参数口径不一致：主控内部枚举 vs 屏端协议值

### 1) 现象

- 工控屏下发路由参数时，界面值与主控内部配置值对不上。
- 典型表现：屏端显示/下发是 `0/1/2`，而主控落盘和脚本切换使用的是 `1/2/3`。
- 结果是 0x04 上报给工控屏的路由字段在部分场景出现偏移，导致“看起来改了，但回显/联动不一致”。

### 2) 原因

- 最终定位为“枚举值口径不一致”：

1. 主控内部（`/etc/node_xwg` 与路由脚本）使用枚举：`1=OLSR, 2=AODV, 3=BATMAN`。
2. 屏端串口协议字段定义使用枚举：`0=OLSR, 1=AODV, 2=BATMAN`。
3. 若把内部值直接写入 0x04 的 `route_protocol` 字段，就会产生偏移（例如内部 1 被屏端解释为 AODV）。

### 3) 怎么排查

- 第一步：核对主控内部路由枚举定义。
- 在 `wg_config.h` 中确认 `KD_ROUTING_OLSR/AODV/CROSS_LAYER = 1/2/3`。
- 第二步：核对串口接收侧路由处理。
- 在 `process_cmd_info(PARAM_CH1_ROUTING_PROTOCOL)` 中确认屏端下发 `0/1/2`，再映射到内部 `1/2/3` 执行脚本与落盘。
- 第三步：核对 0x04 上报封包赋值。
- 在 `Send_0x04` 中定位 `frame_send.Channel1.route_protocol` 的来源，确认是否做了内部值到协议值的转换。

### 4) 最后修改

- 修改文件：`projects/ui_practice_test/mgmt-ctrl/ui_get.c`
- 新增映射函数：`#sym:xwg_route_to_proto`
- 逻辑：将内部路由值 `1/2/3` 转为屏端协议值 `0/1/2` 后再写入 0x04：

```c
static uint8_t xwg_route_to_proto(int router_raw)
{
	/* /etc/node_xwg: 1=OLSR,2=AODV,3=BATMAN; 屏端协议: 0/1/2 */
	if (router_raw < 1 || router_raw > 3) {
		return 0u;
	}
	return (uint8_t)(router_raw - 1);
}

router_raw = get_int_value((void*)param_pairs,"router");
frame_send.Channel1.route_protocol = xwg_route_to_proto(router_raw);
```

- 结果：路由字段在“主控内部值”和“屏端协议值”之间建立了明确转换，联调时回显与行为一致。

## 2026-04-02 Web改工作模式+四频点后出现非法参数值

### 1) 现象

- 在 Web 端将工作模式改为自适应选频，并配置 4 个频点后，主控日志出现明显异常值：

```text
调用mgmt_netlink_set_param函数
[SIM] Received parameter update request.
[SIM] Frequency set to 808923696
[SIM] Tx Power set to 12066
[SIM] Bandwidth set to 53
[SIM] Work mode set to 30839
[SIM] Slot length set to 0
[SIM] Power level set to 100
```

- 这些值超出正常业务范围（例如带宽应在小范围枚举内，工作模式也应为有限枚举值）。

### 2) 原因

- 问题出在“Web/CGI 写库 + sqlite 回调解析”这一步，而不是 netlink 或 SIM 打印逻辑本身。

1. `cmeshInfo.c` 的 POST 处理会把一整组参数都写回 `meshInfo`，即使某字段本次未填写。
2. 对于空字段，CGI 会写入字符串 `NULL`。
3. `sqlite_set_meshinfo_callback` 中多个字段直接 `sscanf(argv[1], "%d", &temp)`，未检查返回值。
4. 当 `argv[1]` 为 `NULL`/非法文本时，`sscanf` 失败，`temp` 保留未初始化值。
5. 该未初始化值被继续写入 `meshinfo`/`mparam` 并下发，最终在 SIM 端看到“随机大数”。

### 3) 怎么排查

- 第一步：核对 CGI 是否会产生空值字符串。
- 在 `cmeshInfo.c` 中确认 `=` 后为空时会手动写入 `NULL`，并且随后统一批量写库。

- 第二步：核对数据库消费回调。
- 在 `sqlite_set_meshinfo_callback` 中检查各字段解析，确认是否对 `sscanf` 返回值做校验。

- 第三步：对照异常日志对应字段。
- 将日志中的 `Frequency/Tx Power/Bandwidth/Work mode` 与回调中的对应赋值路径逐一比对，确认是解析入口污染了下发结构体。

### 4) 最后修改

- 修改文件：`projects/ui_practice_test/mgmt-ctrl/sqlite_unit.c`
- 增加安全解析函数 `parse_mesh_int(name, text, out)`：
1. 统一校验 `text` 非空。
2. 校验 `sscanf` 返回值必须为 1。
3. 解析失败打印告警并跳过该字段，不再继续下发垃圾值。

- 已替换关键字段解析路径：
1. `rf_freq`
2. `m_chanbw`
3. `m_rate`
4. `workmode`
5. `m_select_freq1~4`
6. `m_route`
7. `m_slot_len`
8. `m_trans_mode`
9. `power_level`
10. `power_attenuation`
11. `rx_channel_mode`

- 修复后结果：
1. 非法输入不会再污染 `Smgmt_set_param`。
2. SIM 日志不再出现随机大数。
3. 对于空/非法字段，会出现明确告警并跳过该项下发。

- 补充源头侧修复：`projects/ui_practice_test/www/cgi/cmeshInfo.c`
1. POST 解析从“固定下标 `value[0..17]`”改为“按字段名 key-value 解析”。
2. 仅更新“本次实际提交且非空”的字段，不再批量覆盖整表。
3. 增加 URL decode（`+`、`%xx`），避免编码值误入库。
4. SQL 写入时对 value 做转义，降低异常字符破坏 SQL 的风险。

- 源头修复后的直接收益：
1. 设置 `workmode=4` 时，未填写的 `power_level/power_attenuation/rx_channel_mode` 不会再被空值覆盖。
2. 数据库到主控的参数链路不再因“空字段污染”触发随机非法值。

## 2026-04-02 HMI 选频联调：FIX/ADPT 切换回刷与自适应频点显示滞后

### 1) 现象 A：切到 ADPT 后，进入频率菜单会被刷新回 FIX

- 菜单已按需求实现：FIX 仅显示中心频率，ADPT 显示 F1~F4。
- 实际联调中，选中 ADPT 并进入频率菜单后，界面会被周期刷新成 FIX，导致无法连续配置 4 个频点。

### 2) 原因 A

1. 主控 `write_ui_Thread` 每轮都会发送 0x08（约 1 秒一轮），状态源来自 `/etc/node_xwg` 当前值。
2. 在“ADPT 待配置四频点、尚未最终提交”窗口内，主控侧仍可能上报 `workmode=FIX`。
3. 屏端接收 0x08/0x04 时原先无条件回写 `PARAM_ID_FREQ_HOP_MODE`，本地菜单状态被覆盖。

### 3) 最后修改 A

- 修改文件：
1. `projects/HMI_RTOS_Panel/Core/Inc/app_menu.h`
2. `projects/HMI_RTOS_Panel/Core/Src/app_menu.c`
3. `projects/HMI_RTOS_Panel/Core/Src/app_uart_handlers.c`

- 修复策略：增加“ADPT 编辑保护态”。
1. 在菜单层新增保护态查询/清除接口。
2. 当 HOP MODE 选中 ADPT 且进入“四频点待提交”阶段时，开启保护态。
3. 在 0x08/0x04 接收处理中，保护态期间忽略 `hop=0` 的覆盖，以及 `adapt freq1~4` 的回写。
4. 当收到 `hop=1`（主控确认已处于 ADPT）时自动解除保护，恢复正常同步。

- 结果：ADPT 配置窗口不再被周期上报刷回 FIX，可连续完成四频点编辑。

### 4) 现象 B：四频点已一次性下发，但屏显仍显示上一轮频点

- 复现示例：先 FIX 1450，再切 ADPT 配置 `300/400/500/600`。
- 下发动作完成后，屏显短时仍显示上一轮自适应频点，而不是本轮新值。

### 5) 原因 B

1. 主控 `process_cmd_info(PARAM_CH1_FREQ_HOPPING_MODE)` 在收到 `hop=ADPT` 时会立刻将当前缓存 `s_select_freq1~4` 提交到内核并写 `/etc/node_xwg`。
2. 屏端原先发送顺序是“先发 HOP=ADPT，再发 F1~F4”。
3. 因此主控在处理 HOP 时拿到的是“上一轮缓存频点”，形成“本轮设置成功但屏显滞后一轮”的表现。

### 6) 最后修改 B

- 修改文件：`projects/HMI_RTOS_Panel/Core/Src/app_menu.c`
- 在“四频点掩码满 0x0F 后的一次性提交”中，调整发送顺序为：
1. 先发 `PARAM_ID_ADAPT_FREQ1~4`
2. 后发 `PARAM_ID_FREQ_HOP_MODE=1`

- 结果：主控在切到 ADPT 时读取到的已是本轮最新四频点，屏显不再滞后一轮。

### 7) 联调结论

1. “切换被回刷”本质是屏端延迟提交策略与主控周期状态上报的竞争，需接收侧保护态消抖。
2. “显示上一轮频点”本质是跨端命令时序问题，需保证 ADPT 场景下严格遵循“频点先发、模式后发”。
3. 两项修复叠加后，流程满足需求：选 ADPT 不立即提交，四频点全部确认后整体提交，菜单过程不回刷，回显与本轮设置一致。

## 2026-04-02 Web 回显链路问题：按键已生效但 meshInfo 页面未及时显示

### 1) 现象

- 工控屏按键将模式切到自适应并修改四频点后，主控实际参数已生效。
- 但 Web `meshInfo` 页面仍显示旧模式或旧频点，未及时切换为自适应内容。

### 2) 原因

1. Web 端 `meshInfo.html` 的数据来源是 `cgi-bin/cmeshInfo`，而 `cmeshInfo` GET 仅查询 `meshInfo` 表，不直接读运行态参数。
2. 主控按键链路 `process_cmd_info` 以前未在关键分支同步 `workmode/rf_freq/m_select_freq1~4` 到 `meshInfo` 表。
3. 因此出现“运行态已变更，但数据库仍是旧值”，导致 Web 回显滞后或不一致。
4. 临时在 CGI 侧读取 `/etc/node_xwg` 虽可补齐回显，但会引入跨模块读写耦合与潜在并发竞争，不符合“Web 纯数据库读取”原则。

### 3) 怎么排查

1. 先确认 `meshInfo.html` 读取路径：页面 `GET /cgi-bin/cmeshInfo`，后端执行 `SELECT * FROM meshInfo` 输出 XML。
2. 再对比主控按键处理路径：检查 `ui_get.c::process_cmd_info` 是否在 `PARAM_CH1_FREQ_HOPPING_MODE` 和 `PARAM_CH1_SELECTED_FREQ_1~4` 分支写库。
3. 交叉验证现象：若 `/etc/node_xwg` 已更新但 `meshInfo` 对应行未更新，页面必然显示旧值。

### 4) 最后修改

- 修改文件：
1. `projects/ui_practice_test/mgmt-ctrl/ui_get.c`
2. `projects/ui_practice_test/www/cgi/cmeshInfo.c`

- 修复内容：
1. 在主控 `process_cmd_info` 中补充数据库同步：
2. 模式切换到定频/自适应时同步 `workmode`。
3. 定频中心频率同步 `rf_freq`。
4. 自适应四频点修改时同步 `m_select_freq1~4`。
5. 移除 `cmeshInfo` 中“GET 时读取 `/etc/node_xwg` 回写数据库”的临时逻辑，恢复 CGI 纯数据库读取。

### 5) 结果

1. 数据职责恢复清晰：主控负责参数落库，Web 负责读库展示。
2. `meshInfo` 页面可按数据库实时显示最新 `workmode` 与四频点。
3. 避免了 CGI 直接读取运行态文件带来的读写冲突风险与模块耦合。

## 2026-04-02 0x0A 成员详情工作频率显示为 0：抓帧解码后发现单位不一致

### 1) 现象

- 屏端进入“节点详细信息 -> CH1 -> WORK FREQ”时，字段显示为 `0`。
- 同时成员 ID、IP、跳频方式等其他字段显示正常，说明 0x0A 回包并非整体失败。

### 2) 怎么排查

1. 在主控发送 0x0A 路径打印原始帧字节流。
2. 对抓到的 0x0A 帧逐字段解码，确认工作频率字段原始值。

典型帧样例：

```text
D55D0A000201C0A8020200803E6D560003020C000023365DD5
```

按协议解码后：

1. `D5 5D`：帧头
2. `0A`：命令字（节点详细信息）
3. `00`：消息类型（应答）
4. `02`：成员 ID
5. `01`：空域滤波状态
6. `C0 A8 02 02`：IP（192.168.2.2）
7. `00`：跳频方式
8. `80 3E 6D 56`：工作频率（小端）= `0x566D3E80` = `1450000000`

### 3) 原因

1. 0x0A 帧里主控下发的 `ch_working_freq` 使用了 **Hz**（如 `1450000000`）。
2. 屏端参数字典 `PARAM_ID_DETAIL_CH1_FREQ` 定义范围是 **kHz**（`225000~2500000`，`scale=3`）。
3. 屏端写参时有范围检查，超过上限会拒绝写入，参数保持默认值 `0`。
4. 所以表现为“帧能收到、字段能解码，但工作频率显示为 0”。

### 4) 最后修改

- 主控侧（`projects/ui_practice_test/mgmt-ctrl/ui_get.c`）：
1. 0x0A 测试数据中的 `ch_working_freq` 改为 kHz 口径。
2. `read_xwg_info()` 构造 0x0A 详情时，频率按 kHz 写入，不再乘 1000。

- 屏端侧（`projects/HMI_RTOS_Panel/Core/Src/app_uart_handlers.c`）：
1. 在 `handle_cmd_0x0A` 增加兼容逻辑：若收到值 `>3000000`，判定为 Hz 并自动 `/1000` 转为 kHz。
2. 兼容历史固件与新固件，避免再次出现单位口径不一致导致的显示异常。

### 5) 联调结论

1. 本问题不是 0x0A 串口收发失败，而是“同字段跨端单位定义不一致”。
2. 通过“抓帧 + 解码 + 参数范围对照”可快速定位到单位层面的协议口径问题。
3. 当前方案已同时修复源头口径并加接收兼容，`WORK FREQ` 显示恢复正常。

## 2026-04-02 Web 提交自适应频点后出现非法频率值：触发点在 Web，根因在主控字节序链路

### 1) 现象

- 在 Web 页面仅修改第一个自适应频点（`m_select_freq1`）后，主控日志偶发出现异常频率：

```text
调用mgmt_netlink_set_param函数
[SIM] Received parameter update request.
[SIM] Frequency set to -1157562368
...
调用mgmt_netlink_set_param函数
[SIM] Received parameter update request.
[SIM] Frequency set to 466
```

- 现象表现为“同一轮操作中频率先出现非法大负值/异常值，随后又回到正常范围附近”。

### 2) 怎么排查

1. 沿参数链路逐层定位：`meshInfo.html -> cmeshInfo.c -> sqlite_set_meshinfo_callback -> sqlite_set_param -> mgmt_netlink_set_param -> sim_set_param`。
2. 核对 `rf_freq` 在各层的字节序处理，重点检查是否出现“缓存主机序/网络序混用”。
3. 对照 SIM 端打印口径：`sim_set_param` 读取 `sparam->mgmt_mac_freq` 时会执行 `ntohl`。

### 3) 原因

1. `cmeshInfo.c` 的 POST 行为本身是按字段名更新，并非固定下标错位；它是触发条件，不是根因。
2. 根因在主控 `sqlite_unit.c` 频率链路：
3. `rf_freq` 分支先执行 `htonl` 存入缓存 `meshinfo.rf_freq`。
4. 后续某些分支又把该缓存值直接赋给 `mparam->mgmt_mac_freq`，或通过 `s_fix_freq` 回灌时未统一口径。
5. 下游 `sim_set_param` 再 `ntohl` 一次后，等价于对错误口径做重复转换，最终出现非法值。

### 4) 最后修改

- 修改文件：`projects/ui_practice_test/mgmt-ctrl/sqlite_unit.c`

修复策略：

1. 内部缓存统一使用主机序：`meshinfo.rf_freq`、`s_fix_freq` 都保存主机序频率值。
2. 仅在写入 `mparam->mgmt_mac_freq` 的出口位置执行 `htonl`。
3. 覆盖三处关键出口：
4. `freq_isset` 下发分支。
5. `workmode==1` 的定频回灌分支。
6. 组播同步路径中的频率下发分支。

### 5) 结果

1. Web 修改 `m_select_freq1` 不再触发 `Frequency set to` 异常负值。
2. 参数下发链路口径统一，`mgmt_netlink_set_param` 到 SIM 的频率日志稳定。
3. 问题定性为“后端字节序一致性缺陷”，而非“Web 前端提交格式错误”。
