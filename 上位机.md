# 独立网管软件（上位机）与电台交互协议指南

在整个系统中，设备的配置和监控存在两条并行的通道：
1. **Web 服务器通道（B/S 架构）**：基于无状态的 HTTP 和 CGI 脚本，通过直接读写机器内部的 `test.db` SQLite 数据库完成参数异步下发，不需要与后台进程保活。
2. **专属网管软件通道（C/S 架构）**：运行在电脑（上位机）端的专用网管软件，不经过 SQLite 数据库，而是通过一套基于 **UDP / TCP 套接字** 的自定义管理协议（即包含 `Smgmt_header` 等结构）与电台后端的 `mgmt-ctrl` 进程直接进行网络通信交互。

本文档专门梳理**第二条通道**（上位机通道）中暴露出的网络端口、交互报文类型，以及它们是如何改变电台状态并实现监控闭环的。

---

## 一、 上位机交互核心端口总览（暴露给外部网络）

电台后端的 `mgmt-ctrl` 进程绑定了多个网络端口，上位机需要向特定的端口发送相应的数据包才能穿透到电台管理内核中。以下是直接对上位机端暴露并产生响应的核心监听端口：

| 本地监听端口 | 协议 | 宏定义名 | 核心作用与使用场景 |
| :--- | :--- | :--- | :--- |
| **`16000`** | **UDP** | `CTRL_PORT` / `SOCKET_GROUND` | **上位机专属登录入口与简单透传**。监听基础连接信令和本机的基线参数设置。是建立地空网管最原始的入口。 |
| **`7600`** | **UDP** | `WG_RX_UDP_PORT` / `SOCKET_UDP_WG` | **状态查询、拓扑收集、网络设参网关**。这是承载管理协议最重的一个端口；上位机发起的网内设备探测、拓扑提取、参数配置下发均通过此口。 |
| **`7601`** | **TCP** | `WG_RX_TCP_PORT` / `SOCKET_TCP_WG` | **固件和文件 OTA 升级**。采用 TCP 两段式报文，传输体积较大、对可靠性要求高的二进制升级包。 |
| **`7901`** | **UDP** | `SOCKET_BCAST_RECV` | **基于组播的同步修改**。若上位机想要全网无缝切换（比如瞬间全体换频），应当往组播地址的此端口发包。 |
| **`7903`** | **UDP** | `SOCKET_MEM_REQUEST_RECV` | **底层运行态细项互查（0x0A 协议对接）**。上位机索要非常细致的底层参数或字典时查询的端口。 |

## 二、 报文交互与核心状态机（上位机的“三步走”接管过程）

上位机要实现对整个无线电网络（含本端和深空远端节点）的完美监控，需经过“登录 -> 激活 -> 订阅”三个核心动作。在这个过程中，特定的管理消息会导致电台守护进程的关键标志位发生跳变：

### 1. 登录宣告接管：`MGMT_LOGIN`
* **上位机动作**：向直连的电台网口发送 `CTRL_PORT (16000)` 的 `MGMT_LOGIN` UDP 包。
* **电台内部响应**：
  * 将全局标志位 **`ISLOGIN`** 从 `FALSE` 置为 `TRUE`。
  * 将发送该数据包的上位机 IP 和端口完整记录在 `S_GROUND` 句柄中。
* **意义**：正式宣告“网管软件上线”，电台知晓有真正的 C/S 上位机接入，后续部分被包裹的异常包可以直接透传回这个来源地。

### 2. 状态激活打点：`MGMT_DEVINFO`
* **上位机动作**：向 `WG_RX_UDP_PORT (7600)` 发送 `MGMT_DEVINFO` (0x4d01) 报文，甚至可以指向全网目标 IP。
* **电台内部响应**：
  * 检测到该包针对自身（目标 IP 命中），将全局连接态 **`is_conned`** 置为 `1`。
  * 触发一次主动的 `mgmt_status_report`，向外吐出本机当前的静态信息。
* **意义**：上位机通过模拟 Ping 测试来探测链路上存活的设备；它翻转了电台被管理的连接活跃标识。

### 3. 数据流“订阅”与网关挂载：`MGMT_TOPOLOGY_REQUEST`
* **上位机动作**：向 `WG_RX_UDP_PORT (7600)` 端口发送拓扑请求包 `MGMT_TOPOLOGY_REQUEST` (0x4d0e)。
* **电台内部响应**：
  * 将全局订阅旗标 **`gotRequest`** 置为 `1`。
  * **核心动作**：将发送该请求包的源电脑 IP 地址强制覆盖为自身的 **`gate_addr`**（网管中心网关）。
* **意义**：这一步最重要。一旦收到这个包，直连的这台电台就变成了“路由探针”。这台电台后续通过 `send_topo_msg()` 搜集到底层所有的网络邻居跳数、时延、RSSI 等链路包，都会定向抛给这个 `gate_addr`，从而在上级电脑软件端点亮全网连接拓扑图。

## 三、 上位机承载的核心任务信令 (基于 `Smgmt_header` 规范)

除了接管和心跳，上位机还要发送业务控制命令。业务控制指令同样是以 `Smgmt_header` 开头，具体负载对应不同操作：

### 1. 配置重写 / 下发指令
这些指令大多发送给 `7600` 端口，通过 `Smgmt_data` 承载 `Smgmt_set_param` 内容：
* **本机配置直接下发 (`MGMT_SET_PARAM` / `0x4d02`)**：单台设备的功率、频点、带宽、工作模式变更。报文中包含目标节点 IP；如果该 IP 并非该电台本机，该电台会扮演透明路由转给空中的目标节点。
* **多点网批配置 (`MGMT_MULTIPOINT_SET` / `0x4d0b`)**：指令中包含一个 IP 列表，电台解析后在命中自身的执行配置变更，其余继续广播扔向对应邻居。

### 2. 空中在线升级 (OTA) 指令
这些指令专门通过 **TCP** 连接建立到 `7601` 端口。因为是 TCP 稳态传输，流程分工非常明确，依靠头部的 `mgmt_keep` 来划分传送批次：
* **固件传输准备 (`MGMT_FIRMWARE_UPDATE` / `0x0901` 或 `0x0a01`)**：首先发文件名。电台在 `/root/` 目录下创建对应的空文件句柄。
* **固件二进制追加 (`0x0902` 或 `0x0a02`)**：不断分片发送 `Smgmt_header` 引导的二进制分片数据，电台端使用 `fwrite` 将负载持续写入 Flash。
* **固件接收终了 (`0x0903` 或 `0x0a03`)**：切断写入并释放指针，电台触发 `opkg install` 解包脚本进行系统的热更新，升级完成后尝试清楚安装包缓存。
* **触发独立命令 (`0x0904` 或 `0x0a04`)**：上位机发送纯文本的 Linux Shell Command，电台后端使用 `system()` 执行（等效于给系统留了一个后门级的控制管线）。

### 3. 数据重放与信息透明传输
* **泛数据透传 (`NODEPARAMTYPE` / `0x0001`)**：上位机在 `16001` 等端口推送原始业务报文进行二层/三层透发，电台将其套上一层极简的 `Smgmt_header` 向外转发，不关心内容体。

## 四、 开发调试建议总结
如果你是负责**上位机（C++/C#/Qt 等客户端）的开发对接**，应当遵循以下核心逻辑进行抓包联调：

1. **先抢夺数据上报中心（“挂载”）**：刚接入电台，别急着发控制指令，先向 `7600` 端口发送 `MGMT_TOPOLOGY_REQUEST` 将自身打标为 `gate_addr`，这样才能实时收到这台电台回传的整个空域其它电台健康度，点亮拓扑视图。
2. **区别 Web 逻辑**：如果产品经理问你“为什么网页上点两下就改了，而我的网管软件发包不生效？”——你要清楚，Web 使用的 CGI 就是本地直接写它的 `test.db`，属于强制同步生效；而上位机网络包必须要确保 UDP 头 `Smgmt_header` 的格式完美对其，且大网结构中的 `IP` 设置准确，才能在 `mgmt_recv_msg` 解析链中穿透拦截完成真正生效。
3. **字节序注意**：UDP 通信过程中网络包有大小端转换风险，在写 `Smgmt_set_param` 时注意比如 `uint32_t freq` 等字段的 `ntohl`/`htonl` 操作处理。

## 五、 核心业务流：参数修改下发与实时状态上报机制（代码实现印证）

在电台源码中，针对上位机经常使用的“查看实时曲线 / 拓扑”以及“点对点修改设备参数”两大核心功能，有着极其严密的底层代码印证。以下是这两大业务流在代码中的真实体现逻辑：

### 1. 实时参数上报（电台主动推向上位机）

上位机一旦通过发送 `MGMT_TOPOLOGY_REQUEST` 将自己注册为网管网关（`gate_addr`），电台就会转入“持续喂流”模式。这部分逻辑驻留在电台的 **主轮询回采线程 `mgmt_get_msg`** 中：

* **数据采集源头**：
  `mgmt_get_msg` 线程默认每 5 秒触发一次 `mgmt_netlink_get_info(MGMT_CMD_GET_VETH_INFO)`，通过 Netlink 机制向内核底层驱动索要当前的最新运行态。
* **组装并上报 (`send_topo_msg`)**：
  在拿到底层反馈（含自身状态和周围所有邻居设备的链路连接质量）后，程序会调用 `send_topo_msg(wg_addr, self_msg)`。
* **报文详情 (`MGMT_TOPOLOGY_INFO` / `0x3c03`)**：
  * 电台会构建一个大号的 UDP 包给上位机，消息类型为 `MGMT_TOPOLOGY_INFO`。
  * **负载内容（`mgmt_data`）**：首先包含本机的 `selfip`、`longitude`（经度）、`latitude`（纬度）、底噪、收发总流量。
  * 然后紧跟一个变长的 `neighbor_data[]` 数组。每个邻居占用 12 个字节，分别报告该邻居的 `ID`、`IP` 以及针对该邻居的 **实时信号指标**（如 SNR 信噪比、RSSI 信号强度、排队时延等）。
* **UI 表现**：
  上位机通过在 `7600` 端口持续接收该报文，将其解析后脱壳为坐标图、柱状图或链路连线网状图，给用户“动态刷新监控结果”的视觉反馈。

### 2. 参数修改下发（上位机点控电台）

当用户在上位机软件的界面上，选中网络某一个终端并修改它的核心射频参数（例如：将 192.168.2.2 的带宽修改为 10M）时，以下动作将在后端精准展开：

* **报文触达 (`mgmt_recv_msg` 接收)**：
  上位机构建一个 `MGMT_SET_PARAM` (`0x4d02`) 包抛到目标电台（或随便一个网络内的中继电台）的 `WG_RX_UDP_PORT (7600)` 端口。
* **命中判定与路由提取**：
  在代码分支 `case MGMT_SET_PARAM:` 中，电台提取 `mgmt_data[0..3]`（即目标设备的 IP 地址 `dst_ip`）：
  * 如果 `dst_ip != SELFIP`：电台原封不动地把该包继续扔向空中网络进行透传（相当于上位机的指令可以跨越无线链路在漫游电台间穿梭跳跃）。
  * 如果 **`dst_ip == SELFIP`**：这就表明包送给的就是本地机器，开始执行落地操作。
* **归一化落地设参 (`mgmt_netlink_set_param_wg`)**：
  当判定包就是发给自己时，核心会调用该函数吞下剩余的 `Smgmt_param` 数据。
  * 它首先会解析传入参数，**同步去复写电台本地的 `/etc/node_xwg` 静态配置文件**（确保这台电台未来要是掉电重启，依然保留上位机刚刚给它下发的新频点、带宽配置）。
  * 然后，将修改项组装成底层的变长 Netlink 属性，真正调用系统的通信接口敲击内核，促使设备的物理网卡/射频芯片真正切频、变阵。
* **闭环**：
  此项工作执行完毕后，底层工作模式突变了；下一秒 `mgmt_get_msg` 回采到底层状态后，就会顺理成章把变更后的新状态通过 `MGMT_TOPOLOGY_INFO` 吐回给上位机，这就形成了一个完美的下发到展现的管理闭环。
