# 系统功能验证方案

本文档详细描述了imx6ull开发板心跳数据模拟系统的验证流程，分为三个独立的测试阶段。

## 测试环境配置

*   **硬件平台**: IMX6ULL 开发板
*   **软件环境**: Linux (自定义构建), 模拟心跳程序 (`ui_practice`)
*   **网络环境**: 开发板与测试PC处于同一局域网
*   **工具**: 浏览器 (Chrome/Edge), SSH 终端, SQLite 命令行工具 (可选)

---

## 第一阶段：网管界面实时刷新验证

**目标**: 验证网管界面是否能够根据模拟的心跳包数据实现自动刷新，确保数据的实时性和准确性。

### 测试步骤

1.  **启动模拟程序**:
    *   在开发板终端运行程序: `./ui_practice`
    *   确认终端打印 `[SIM] Simulation initialized.`

2.  **访问网管界面**:
    *   在PC浏览器访问 `http://<开发板IP>/systemInfo.html` (或相关页面)
    *   观察页面上的参数显示。

3.  **观察自动刷新**:
    *   保持页面打开，不进行手动刷新。
    *   观察以下参数是否每隔几秒（通常5秒）自动变化：
        *   **TX/RX 流量**: 应该持续增加。
        *   **信号强度 (RSSI)**: 应该在一定范围内（如 -60dBm 左右）微小波动。
        *   **信噪比 (SNR)**: 随RSSI变化。
        *   **邻居列表**: 应该显示预设的邻居节点（如 ID 2, 3, 4）。

4.  **验证数据一致性**:
    *   在开发板终端查看日志，或者使用 SQLite 工具查询数据库：`sqlite3 /www/cgi-bin/test.db "select * from systemInfo;"`
    *   对比数据库中的值与网页显示的值。

### 预期结果

*   网页参数自动刷新，无需手动F5。
*   流量数据单调递增。
*   信号质量数据呈现合理的随机波动，模拟真实环境。
*   长时间运行（>1小时）页面无卡顿，浏览器内存占用稳定。

### 问题记录

*   *在此记录测试中发现的任何异常，例如刷新停滞、数据跳变异常等。*

## 第二阶段：网管下发功能验证

**目标**: 验证网管系统参数下发功能的完整性，确保配置能正确传递到底层模拟器并反馈回界面。

### 测试步骤

1.  **修改频率参数**:
    *   在网管界面“系统配置”或“宽带配置”页面，修改工作频率（例如从 580MHz 改为 520MHz）。
    *   点击“设置”或“下发”。

2.  **检查日志与状态**:
    *   查看开发板终端日志，应出现：
        *   `[SIM] Received parameter update request.`
        *   `[SIM] Frequency set to 520000`
    *   检查 `/etc/node_xwg` 文件是否被修改：`cat /etc/node_xwg | grep channel`

3.  **验证界面反馈**:
    *   等待几秒钟（心跳周期），观察网管界面上的“当前频率”显示。
    *   **关键验证点**: 界面显示的频率应自动更新为 520MHz，证明数据闭环打通（Web -> CGI -> App -> Sim -> DB -> Web）。

4.  **并发测试**:
    *   尝试快速连续下发不同的参数（如同时修改频率、带宽、发射功率）。
    *   观察系统是否能正确处理所有请求，最终状态是否与最后一次操作一致。

### 预期结果

*   下发指令后，终端立即打印修改日志。
*   配置文件 `/etc/node_xwg` 内容正确更新。
*   网管界面在下一个刷新周期自动显示新的参数值。
*   系统在并发操作下不崩溃，数据最终一致。

## 第三阶段：工控屏与GPS功能验证

**目标**: 验证工控屏参数传递的准确性及GPS数据的接收处理。

### 测试步骤

1.  **工控屏参数显示**:
    *   (如果连接了工控屏) 观察屏幕显示的频率、功率等参数。
    *   在网管界面修改参数，验证工控屏是否同步更新。

2.  **GPS数据模拟/接收**:
    *   由于在室内可能无真实GPS信号，程序可能使用模拟GPS数据或最后一次有效位置。
    *   验证 `gpsget.c` 模块是否正常运行（通过日志或线程状态）。
    *   在网管界面查看“经度”、“纬度”信息。
    *   **模拟测试**: 可以手动修改代码或数据库中的经纬度，验证界面是否更新地图或坐标显示。

3.  **异常处理**:
    *   模拟断开GPS天线（如果使用真实GPS）或停止GPS数据流。
    *   验证系统是否报告“GPS信号丢失”或保持最后已知位置，而不是显示错误代码或崩溃。

### 预期结果

*   工控屏数据与网管界面、底层配置保持三方一致。
*   GPS坐标在界面正确显示。
*   异常情况下系统运行稳定。
